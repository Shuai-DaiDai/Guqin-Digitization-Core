# Jianzi-Code 版本策略

这份策略说明 `Jianzi-Code` 作为独立项目时，应该怎样升级版本、怎样判断变更是否安全，以及文档和示例如何保持同步。

## 基本原则

- 版本号是对外承诺，不只是文件名。
- 任何改动都要先考虑旧数据还能不能读、旧工具还能不能用。
- 规范文档、示例文件和校验工具必须一起更新，不能只改一处。

## 当前版本命名

当前仓库使用以下命名方式：

- 单音事件规范：`jianzi-v1.schema.json`
- 整曲文档规范：`jianzi-document-v1.schema.json`
- 整曲文档版本字段：`schema_version = "jianzi-document-v1"`

这表示项目现在处于 `v1` 体系中。

## 什么算“可加性变更”

可加性变更的意思是：新版本增加了信息，但不破坏老数据的基本理解。

典型情况包括：

- 新增一个可选字段
- 新增一个新的枚举值，但旧值继续保留
- 新增一个示例文件
- 新增一个辅助文档或解释页
- 新增一个不会影响旧字段含义的桥接字段

但要注意：因为当前 schema 对字段控制比较严格，新增字段必须同时更新 schema、示例和说明文档，不能只在数据里“悄悄加上”。

## 什么算“破坏性变更”

破坏性变更的意思是：旧数据、旧工具或旧理解方式会失效，或者会得到不同结果。

典型情况包括：

- 删除字段
- 把原本必填字段改成可选，或者反过来
- 改变字段含义
- 改变字段类型
- 改变枚举值的拼写或语义
- 修改 `id`、`schema_version`、`notation_systems` 这类稳定锚点的规则
- 让旧例子无法通过新校验，且没有迁移方案

这类变更必须升级大版本。

## 版本怎么升

### 1. 小改动

如果只是补充说明、修正文案、增加示例解释，不改数据契约本身，可以不升规范版本，只更新文档。

### 2. 可加性更新

如果新增的是兼容字段或兼容枚举，并且不会破坏已有数据的读取逻辑，可以在 `v1` 体系内发布新的小版本内容。

对整曲文档来说，这通常意味着：

- 继续使用 `schema_version = "jianzi-document-v1"`
- 保持旧字段不变
- 新增内容必须有默认可理解的行为

### 3. 破坏性更新

如果变更会让旧数据不再符合新规范，或者会改变已有数据的解释方式，就必须进入 `v2`。

对当前项目来说，破坏性更新应该表现为：

- 新 schema 文件，例如 `jianzi-v2.schema.json`
- 新文档版本，例如 `jianzi-document-v2.schema.json`
- `schema_version` 变成新的字符串
- 示例文件也换成对应版本

## `schema_version` 的规则

- `schema_version` 只用于整曲文档层。
- 它必须和实际 schema 文件版本一致。
- 一旦进入 `v2`，旧的 `v1` 版本不应该被覆盖掉，而应该继续保留。
- 新旧版本可以并存一段时间，直到迁移完成。

## 示例文件怎么跟着走

示例文件不是装饰品，它是规范的一部分。

每次版本更新时，至少要做到以下几点：

- 示例必须能通过当前校验
- 示例里的字段必须和文档里的术语一致
- 示例要覆盖最重要的典型情况
- 如果增加了新字段，示例里要给出真实可读的写法
- 如果删除或更改了字段，必须同步更新说明页里的讲法

## 文档怎么跟着走

文档应该和规范同步更新，至少包括：

- 字段词典
- 版本策略
- 面向非程序人员的说明页
- 示例文件说明

如果 schema 改了，但文档还停留在旧说法，这会直接伤害协作效率，所以不允许长期悬空。

## 推荐发布节奏

建议把每次发布分成三类检查：

1. 数据检查：示例是否通过校验。
2. 语义检查：字段词典和说明页是否仍然准确。
3. 兼容检查：旧数据是否还能读、旧工具是否还能理解。

只要其中一项不通过，就不要把这次变更当作正式版本发布。

## 一句话规则

能不破坏旧数据就不升大版本；一旦会破坏旧数据或旧理解，就明确进入新版本，并同时更新 schema、示例和说明文档。
