# OCR 输出到 Jianzi-Code 草稿的桥接说明

这份说明不是给程序员看的实现细节，而是给标注、打谱、OCR 和规范协作各方看的“中间接力规则”。

它要回答的是同一件事：

> 一张古琴谱影印页，怎么一步步变成可校对、可追溯、可导出的 `Jianzi-Code` 草稿。

## 为什么要有这条桥

OCR 看到的是图像。

`Jianzi-Code` 记录的是结构化知识。

中间如果没有一条清晰的桥接链路，图像识别、人工校对、演奏理解和最终数据导出就会彼此断开，最后很难复用，也很难比较版本。

所以这条桥的目标不是“让机器一次性全认对”，而是把每一步都拆开：

- 哪一步只负责看见
- 哪一步只负责提出候选
- 哪一步只负责归一化
- 哪一步已经开始接近最终事件
- 哪一步必须由人确认

## 总体链路

我们把整条链路分成六层：

1. `page image`
2. `detection box`
3. `glyph candidates`
4. `normalized note`
5. `event draft`
6. `Jianzi document`

这六层不是六个完全独立的文件格式，而是同一份数据从“图像”走向“规范文档”的六个状态。

每一层都要尽量保留上游信息，不要过早丢掉证据。

---

## 1. Page Image：页图层

### 这一层是什么

这一层是原始影印页或裁切页本身。

它是所有后续判断的出发点，也是回溯问题时最重要的证据。

### 这一层要保留什么

至少保留：

- 页图文件本身
- 页码或页序
- 来源书名、卷名、版本
- 图像分辨率、裁切状态、旋转状态
- 文件名或稳定 ID
- 这页属于哪首曲子、哪一段

如果是整本书里的目录页、封面页、页眉页、题记页，也应保留为页面级来源信息，但不一定进入正文事件链。

### 这一层必须人工确认什么

- 这页是不是正文谱页
- 这页属于哪一首曲子、哪一卷、哪一段
- 页序有没有错位
- 图像有没有翻页、缺角、重影、倾斜、裁切过度等问题

### 这一层的判断原则

页图只负责“把来源留住”，不负责解释减字。

不要在这一层把正文、目录、封面、题记混成一类。

---

## 2. Detection Box：检测框层

### 这一层是什么

这一层是模型或人工在页图上圈出的候选框。

它回答的是最基础的问题：

> 哪一块像一个谱字、一个标题、一个正文单元、一个噪声块？

### 这一层要保留什么

至少保留：

- 框的位置
- 框对应的页图 ID
- 框的类别
- 框的置信度
- 框是否来自模型、人工或回灌结果
- 框与页图之间的空间关系

如果一个框后来被判定为不该进入训练或导出，也要保留它的来源和判断记录。

### 这一层必须人工确认什么

检测框阶段，人工确认的是“框得对不对”，不是“音认得对不对”。

重点确认这些问题：

- 这个框是不是一个完整的减字单元
- 有没有把两个独立减字框进同一个框
- 有没有只框住半个字
- 有没有把目录、页码、封面文字、说明文字、简谱等非正文内容框进去
- 有没有漏掉应该框出来的正文减字

### 这一层的判断原则

框选层看的是“视觉单元”，不是“音乐句法”。

也就是说：

- 一个主音减字是一个框
- 后面单独写出的装饰、后续处理、附加记号，如果在纸面上是独立记号，也应分别成框
- 不能因为音乐上属于同一个音，就强行把几个独立写出来的记号并成一个大框

### 这一层常见的人工处理规则

- 两个减字被一个框一起罩住：判错
- 夹杂简谱、页码、目录文字：判错
- 只要框住的是一个完整独立减字：可判对
- 一页里有漏框：先在备注中记录，后续再补检索或补框流程

---

## 3. Glyph Candidates：谱字候选层

### 这一层是什么

这一层不再只是“框在哪儿”，而是“这个框里像什么字、像哪些部件、像哪些组合方式”。

它通常来自：

- OCR 模型的字形识别
- 部件/偏旁识别
- 规则或字典匹配
- 人工修订时的候选回填

### 这一层要保留什么

至少保留：

- 原始裁切图或裁切引用
- 候选字形
- 候选部件
- 候选顺序
- 每个候选的置信度
- 候选来自哪里：模型、字典、规则、人工
- 与原检测框的绑定关系

如果同一个框有多个可能结果，也要保留多个候选，不要只留一个“猜测答案”。

### 这一层必须人工确认什么

这一层人工确认的是“认字方向对不对”。

重点确认：

- 候选是不是这个减字本体
- 部件拆分是否合理
- 有没有把别的字形误当成这个字
- 是否需要保留多个候选供后续再看
- 候选是否已经足以进入规范化阶段

### 这一层的判断原则

候选层不是最终真值层。

它的作用是把“一个框里可能是什么”尽量保留完整，让后面的规范化可以在更少损失的前提下继续。

---

## 4. Normalized Note：归一化音符层

### 这一层是什么

这一层开始把谱字候选整理成 `Jianzi-Code` 里的标准音符草稿。

它不再只是“像什么字”，而是要把字形、指法、弦位、音类这些信息对齐成结构化字段。

### 这一层要保留什么

至少保留：

- 该条记录来自哪个页图、哪个框、哪个候选
- 视觉层信息
- 物理层信息
- 音响层信息
- 每个字段的来源与置信度
- 哪些字段是推断出来的，哪些字段是人工确认的

### 这一层通常会整理出的字段

视觉层通常包括：

- 完整字形兜底
- 结构布局
- 部件槽位

物理层通常包括：

- 弦号
- 徽位与分位
- 右手手指和技法
- 左手手指和动作
- 散音、按音、泛音等类型

音响层通常包括：

- 音名
- MIDI 音高
- 时值
- MusicXML 片段

### 这一层必须人工确认什么

归一化层里，必须人工确认的通常是这些：

- 这是不是一个完整独立的减字事件
- 弦号是否正确
- 徽位和分位是否正确
- 右手技法是否正确
- 左手动作是否正确
- 这是散音、按音还是泛音
- 这里有没有把主音和后续装饰混在一起
- 音高和时值是否应该保留、修正或留空

### 这一层的判断原则

如果上游只是“可能看对了字”，这里就不能假装已经完全定稿。

凡是低置信度字段，或者语义上依赖上下文的字段，都应显式保留“待确认”状态。

---

## 5. Event Draft：事件草稿层

### 这一层是什么

这一层是已经接近 `Jianzi-Code` 的整条记录草稿。

它把归一化后的音符，连同来源、版本、信心与待办项一起打包成一条可审阅事件。

### 这一层要保留什么

至少保留：

- 事件 ID
- 原始页图来源
- 原始检测框
- 候选列表
- 归一化结果
- 缺失字段
- 低置信度字段
- 是否已回灌或已排除

这一层的核心价值是可追溯。

当草稿被后来改动时，仍然能追到：

- 它原来来自哪页
- 先前是什么框
- 哪个候选被采用了
- 哪个字段是人工补的

### 这一层必须人工确认什么

事件草稿层必须人工确认的内容，通常是：

- 这条草稿能不能正式进入最终文档
- 哪些字段要补
- 哪些字段要改
- 哪些草稿其实应该删掉
- 主音、装饰、后续处理之间的边界是否处理正确

### 这一层的判断原则

事件草稿已经不是纯 OCR 中间结果了。

它是“可校对的半成品”。

因此这里的目标不是追求看起来像最终答案，而是让所有未确定的地方都明确可见。

---

## 6. Jianzi Document：整曲文档层

### 这一层是什么

这一层是最终面向保存、交换和导出的整曲结构。

它会把事件草稿放进曲目级上下文里，形成完整的 `Jianzi-Code Document`。

### 这一层要保留什么

至少保留：

- 曲名
- 来源书名、版本、卷次
- 定弦
- 段落、小节、顺序
- 每条事件的稳定 ID
- 事件之间的前后关系
- 版本来源和导出时间

### 这一层必须人工确认什么

最终文档层必须人工确认的，通常不只是单个字，还包括整曲结构：

- 事件顺序是否正确
- 段落和小节边界是否合理
- 有没有重复或漏掉的事件
- 目录页、封面页、页码页、说明页是否被误纳入正文
- 主音与后续装饰是否按正确顺序连接
- 整曲是否可以导出给后续播放、研究或比较使用

### 这一层的判断原则

最终文档不是把所有候选硬拼在一起，而是把经过确认的事件，按可追溯的结构稳定保存下来。

---

## 哪些地方必须人工确认

如果只记住一件事，就记住这一条：

> 人工确认永远不能只留在最后一步，必须分布在链路中间。

最少要人工确认的环节是：

1. 页面是不是正文页
2. 框是不是一个完整谱字单元
3. 谱字候选是不是认对了
4. 归一化字段是不是补对了
5. 草稿是不是可以进入最终文档
6. 整曲结构是不是排序正确

如果某一步明显不确定，就不要假装它已经自动完成了。

---

## 推荐的保存原则

为了让这条链路长期可用，建议遵守四个原则：

- 原始图像不改，只引用
- 框和候选都要保留来源
- 低置信度不自动抹掉，要显式标出
- 人工确认结果要能回写到上游和下游

这意味着：

- OCR 不是一次性的
- 校对不是末尾补丁
- 草稿不是废弃中间态
- 每一步都应能回看

---

## 这条桥的实际用途

这条桥最重要的作用，是让不同角色能在同一条链路上分工：

- 视觉识别的人负责给出页图上的候选框
- 识字的人负责确认谱字和部件
- 懂演奏的人负责确认弦位、指法和音类
- 懂数据的人负责把它们稳定落到 `Jianzi-Code` 草稿里

这样一来，大家讨论的就不再是“这个图大概像什么”，而是“这条链路的哪一步还需要确认”。

---

## 一句话总结

`page image -> detection box -> glyph candidates -> normalized note -> event draft -> Jianzi document`

这条链路的本质，不是把图像尽快变成答案，而是把每一步都保留下来，让它们可以被复核、补全、追溯和交换。

如果某一步必须人看，就不要把它藏起来。

如果某一步只是候选，就不要把它写成定稿。

如果某一步已经确认，就把它稳稳地交给下一步。
