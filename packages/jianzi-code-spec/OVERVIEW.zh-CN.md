# Jianzi-Code 是什么

`Jianzi-Code` 是一套面向古琴减字谱的通用数字规范。

它的目标，不是把古琴谱“翻译成程序员才能看懂的东西”，而是把古琴谱里原本压缩在一个减字里的信息，拆解成一份能够长期保存、反复校对、稳定传递的标准记录。

可以把它理解成一张“标准化信息卡”：

- 它保留这个减字原来长什么样
- 它说明演奏者应该怎么弹
- 它记录最后应该发出什么音

这样一来，同一个谱字就不再只是书页上的一个字，而变成一个可以被检索、比较、校对、分析、播放和再利用的数据单元。

## 它解决的是什么问题

今天古琴谱数字化，常常卡在三个地方：

1. 图像识别只能“看到字”，却不知道这个字内部是什么意思。
2. 音乐转写往往只保留最后的音高和时值，却丢掉了古琴真正重要的指法和按位信息。
3. 不同团队使用不同的录入方法，最后很难交换、合并和长期积累数据。

`Jianzi-Code` 的作用，就是在这三者之间搭一座桥。

它既能接住 OCR 识别的结果，也能接住人工校订的演奏理解，还能继续往下生成 MusicXML、MIDI 或其他分析材料。

换句话说，它不是某一个软件的私有格式，而是一个“中间标准”。

## 为什么它对音乐专业的人有价值

这份规范最重要的价值，不是“让机器看懂古琴谱”，而是让不同背景的人终于能围绕同一份谱面说同一种话。

对古琴演奏者和研究者来说，它的价值在于：

- 可以把减字中的演奏动作单独保存下来，不会在转成简谱或五线谱时被抹掉。
- 可以把同一首曲子的不同版本、不同打谱理解并排比较。
- 可以把“这个字怎么认”“这个字怎么弹”“这个音该怎么落”分开讨论，而不是混在一起。
- 可以把过去只能写在批注里的经验，变成可积累、可检索、可共享的数据。

对将来的整理、出版、教学和研究来说，这意味着：

- 曲谱校勘会更清楚
- 指法研究会更细
- 数字乐谱会更可靠
- 跨项目合作会更容易

## 这份规范到底记录什么

`Jianzi-Code` 的核心思想很简单：一个音，分三层记录。

### 第一层：它长什么样

这一层记录减字本身的视觉信息。

比如：

- 这个字完整写出来是什么
- 它是上下结构、左右结构，还是更复杂的组合
- 它的几个关键部件分别落在哪个位置

这一层主要服务于：

- OCR 识别
- 字形分析
- 前端排版展示
- 人工校对“识别对不对”

### 第二层：它怎么弹

这一层记录演奏动作信息。

比如：

- 弹的是第几弦
- 按在什么徽位、徽分
- 右手用什么手指、什么技法
- 左手用什么手指
- 这是散音、按音，还是泛音
- 有没有吟、猱、绰、注、滑音等附加动作

这一层是古琴谱最不能丢的部分。很多别的数字格式只记录最后发出的音，但古琴谱真正的知识密度，其实主要就在这一层。

### 第三层：它听起来是什么

这一层记录现代音乐系统更容易直接处理的结果。

比如：

- 音名
- MIDI 音高
- 时值
- 对应的 MusicXML 片段
- 如果需要，也可以保留简谱信息

这一层主要服务于：

- 回放
- 乐谱导出
- 和简谱、五线谱、MIDI 的互通
- 机器分析

## 它不是要取代什么

`Jianzi-Code` 不是要取代减字谱本身。

它也不是要把所有人都变成程序员，更不是要求研究者直接去写代码。

它的定位更像：

- 对谱面图像来说，它是解释层
- 对人工校订来说，它是记录层
- 对播放器和分析工具来说，它是交换层

所以它和简谱、五线谱、MusicXML、MIDI 的关系，不是谁替代谁，而是谁负责保存哪一部分信息。

减字谱保存传统记谱方式。
`Jianzi-Code` 保存结构化知识。
MusicXML、MIDI 保存播放与现代乐谱输出。

## 音乐专业的人将来怎么使用它

最常见的工作方式会是下面这样：

1. 先拿到一页古琴谱影印图。
2. 系统先识别出谱字，并给出初步结果。
3. 音乐专业的人查看每个字，确认这个字认得对不对。
4. 然后再确认它属于散音、按音还是泛音，应该落在哪根弦、哪个徽位，用什么指法。
5. 最后，如果需要，再把它导出成可播放的乐谱或音频材料。

在这个过程中，音乐专业的人并不需要关心技术实现。

你真正需要做的，是对着一条条记录回答熟悉的问题：

- 这个字认对了吗？
- 这个部件拆对了吗？
- 这根弦对吗？
- 这个徽位合理吗？
- 这里应该是泛音还是按音？
- 这里的右手技法该怎么记？

也就是说，这份规范不是让你改行，而是让你能把专业判断更准确地留下来。

## 一条记录大概长什么样

从人的角度看，一条记录其实就是这样：

- 这是哪个音
- 它来自哪个减字
- 这个减字是什么结构
- 演奏时用哪根弦、哪种指法、哪种音类
- 最终发出什么音、持续多久

如果将来在界面里展示，它完全可以不是一段代码，而是一张结构化表单。

例如可以被展示成：

| 项目 | 内容 |
|---|---|
| 减字 | 泛 |
| 音类 | 泛音 |
| 弦位 | 三弦 |
| 徽位 | 七徽半 |
| 右手 | 勾 |
| 左手 | 不按 |
| 音高 | C3 |
| 时值 | 一拍 |

所以这份规范本质上是在定义“一张表单该有哪些栏位”，而不是逼人去读程序。

## 为什么它适合作为独立项目

`Jianzi-Code` 值得做成独立项目，因为它不是某一个模型、某一个网站、某一个数据集的附属品。

它有机会成为更基础的公共层：

- 上面可以长出 OCR 工具
- 上面可以长出人工校订平台
- 上面可以长出数字乐谱导出工具
- 上面可以长出教学、检索、比较和研究应用

如果没有这样一层共同规范，每个项目都会重复造轮子，而且彼此之间很难互认数据。

如果有了这层规范，大家即使使用不同工具，最后也能汇总到同一套标准里。

## 我们希望它带来的合作方式

将来这套规范最理想的合作方式，是让不同背景的人各自贡献最擅长的部分：

- 做图像和模型的人，负责把谱面尽可能识别出来
- 懂减字结构的人，负责判断字形和部件
- 懂演奏和打谱的人，负责确认指法、徽位和音类
- 做数字音乐的人，负责导出 MusicXML、MIDI 和回放结果

大家不是各做各的，而是在同一份结构化记录上接力工作。

## 一句话总结

`Jianzi-Code` 不是“给机器看的古琴谱”。

它更准确的定义应该是：

一套把古琴减字谱中的视觉信息、演奏信息和音响信息同时保存下来的公共数字标准。

它的意义，不只是方便技术开发，更是为了让古琴谱的知识能够被长期整理、校对、比较、传播和继承。
